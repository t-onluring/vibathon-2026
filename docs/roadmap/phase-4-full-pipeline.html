Pipeline Steps

Nine steps organized into three subsystems: extraction for new content types, infrastructure for reliability and performance, and consumer-facing features for API access and quality.

Extraction
01
Video content extraction
Extract event info from video thumbnails and captions. Most videos are forwarded lecture recordings with event details in the thumbnail or caption text.
video-extractor.ts, video-downloader.ts
02
Forwarded message tracking
Track forward_from and forward_from_chat metadata. Link events back to original source channel. Detect cross-posts across topics.
forward-tracker.ts
Infrastructure
03
Cross-source deduplication engine
Deduplicate events appearing in multiple sources. Match by date + masjid + ustadz with fuzzy matching. Keep the richest version with the most populated fields.
dedup-engine.ts
04
Near real-time processing
Switch from 6-hour polling to Telegram webhook or 15-minute polling. Process new messages incrementally. Hybrid approach: webhook for primary, hourly cron as safety net.
webhook-handler.ts
05
Database migration
Evaluate moving from JSON files to a database. Options include SQLite via Turso/LibSQL, Supabase PostgreSQL, or staying with JSON. Decision hinges on whether consumer apps need dynamic queries.
migration scripts (if needed)
Consumer
06
Public API with documentation
Formalize the API with versioning, rate limiting, and OpenAPI spec. Eight endpoints covering sources, events, health, and metadata.
generate-api.ts, src/app/api/
07
Event verification workflow
Allow community members to verify and correct extracted events. Show extracted data alongside raw message for confirm, edit, or reject actions.
VerifyTab.tsx
08
Multi-platform source ingestion
Extend beyond Telegram: website scrapers for jadwalkajian.com and portalkajian.online, WhatsApp forward-to-bot bridge, YouTube RSS feeds, Instagram scraping.
per-platform ingestion modules
09
Monitoring and alerting
Pipeline health dashboard tracking ingestion success rate, extraction accuracy, and event counts. Alerting for disconnections, accuracy drops, and cost overruns.
monitoring module
03

API Endpoints

Eight versioned REST endpoints providing event discovery, source management, and operational metadata.

Endpoint Description
GET /api/v1/sources List all registered sources
GET /api/v1/sources?region={region} Filter sources by region
GET /api/v1/events?date={date} Events on a specific date
GET /api/v1/events?region={region} Events filtered by region
GET /api/v1/events?ustadz={name} Events by speaker name
GET /api/v1/events?lat={lat}&lng={lng}&radius={km} Nearby events by geolocation (requires DB)
GET /api/v1/health Pipeline health check status
GET /api/v1/meta API metadata and aggregate statistics

Risks

Key risks ranked by severity, with proposed mitigations for each.

Risk Severity Mitigation
Dedup false positives (different events merged) High Conservative matching thresholds, manual review queue
Webhook hosting complexity Medium Start with frequent polling, migrate to webhook later
Multi-platform scrapers break on site changes Medium Per-source health monitoring, graceful degradation
Cost escalation with more sources Medium Tiered processing: high-priority sources only for vision
Community verification adoption Low Start with admin-only verification
